# CodeAxis 使用教程

CodeAxis 是信川机械工业内部的 AI 代码助手聊天工具：帮工程师浏览/搜索自己有权限的仓库、确认 bug、逐步讲解代码，并能直接从对话里起草 GitHub/GitLab issue。

**它是只读的**——不会改代码、不会直接提交补丁。它确认问题、解释代码、起草一份 issue，剩下的交给人来做。

访问地址：**https://codeaxis.852098505.xyz/**

## 一、管理员：首次初始化

### 1. 登录

打开 `/login`，用户名密码登录。**首次部署默认账号是 `admin` / `admin123`，登录后请立刻去「用户」页给 admin 改一个真实密码**——这是明摆着的默认口令，别留着。

### 2. 添加仓库（「仓库」标签页）

进「仓库」标签页，填「添加仓库」表单：

| 字段 | 说明 |
|---|---|
| 仓库名称 | 展示用的名字，随便起 |
| Git URL | `https://` / `http://` / `git://` 开头的 clone 地址 |
| 分支（可选） | 留空则用远程默认分支 |
| 用户名（私有仓库可选） | 走 HTTP Basic |
| 令牌/密码（可选） | GitLab/GitHub 的 PAT，或密码 |
| 描述（可选） | 备注 |

点「添加」后，仓库会在后台自动 clone + 建索引（ctags 符号索引 + 语义检索 embedding），「同步状态」那一栏能看到进度。首次索引一个仓库可能要几分钟，之后每次同步只处理增量。

> 出于安全考虑，仓库 URL 里**不要**直接拼 `https://user:token@host/repo.git` 这种带凭证的写法——凭证走专门的「用户名」「令牌/密码」字段，这样其它人在界面上看到的 URL 里不会带凭证明文。

### 3. 建用户 + 授权（「用户」「权限」标签页）

「用户」页创建账号（用户名、密码≥8位、角色选「普通用户」或「管理员」）。

「权限」页把「用户」和「仓库」配对，权限级别选：`只读` / `读写` / `管理`。**目前这三个级别在功能上完全等价**——不管选哪个，实际拿到的都是同样的只读工具访问（读代码、搜索、看符号定义），代码里还没有按级别做区分，纯粹是个预留字段。真正起作用的是「有没有被授权」这个二元判断：**普通用户只能看到被明确授权过的仓库**，没授权的仓库对他们完全不可见。

## 二、日常使用（普通用户视角）

### 1. 新建会话

左上角「+ 新建会话」。左侧「Workspace」区是被授权的仓库列表，点一下某个仓库的 chip 把它选中（一次只能选一个，选中状态会持续到你换仓库或换会话）——每条消息都会带着这个 `repo_id` 发给后端，工具只会在这个仓库范围内搜索/读取。

### 2. 提问

直接在输入框描述你要做的事。首页给了三个示例，可以直接点：

- 「这段代码逻辑是什么？帮我逐步讲解」
- 「确认一下这个 bug 是否真的存在」
- 「帮我起草一个 GitHub issue」

支持贴图（截图报错信息很方便）：点输入框左边的 `+`，或者直接 `Ctrl+V` 粘贴/拖拽图片进来，最多 5 张。`Enter` 发送，`Shift+Enter` 换行。

### 3. 看它在做什么

回答里出现的工具调用（读文件、搜代码、查符号定义、语义检索）会折叠成一个可展开的记录条，点开能看到具体调用了什么、返回了什么，不是黑箱。回答里提到 `文件名:行号` 这种引用，点一下会在右侧弹出只读代码查看面板（`Esc` 关闭）。

### 4. 给回答打分

每条回答下面有 `▲ 有帮助` / `▼ 不准确` 两个按钮，顺手点一下——管理后台「用量」页会汇总最近的差评，这是让这个工具越用越准的唯一反馈渠道。

## 三、核心工作流：确认 bug → 起草 issue

这是这个工具存在的核心理由，流程是这样的：

1. 描述你怀疑的问题（比如「确认一下这个 bug 是否真的存在」+ 具体现象）。
2. 模型会去读代码、跑必要的搜索，确认问题是否真实存在。
3. 确认后它会调用起草 issue 的能力，界面上出现一张 **issue 草稿卡片**：标题、（如果适用的）期望行为、正文、标签。
4. 提交前会自动查一遍这个仓库里有没有相似的 issue，如果有会提示「发现 N 个相似 issue，提交前请确认不是重复」，并标出每条是「开放中」还是「已关闭」——**先看一眼再提交**，避免刷重复单。
5. 确认没问题，点「确认提交」。提交成功后这个会话会标记为「已完结」（✅ 本次任务已完结 — 发送新消息将开始一个新会话），发下一条消息会自动开一个新会话，不会接着往这条已经提交的 issue 对话里加内容。

如果不想提交，点「取消」就行，不会有任何副作用。

### 管理已有 issue

如果问的是「给某个已有 issue 加一条评论/关掉它/重新打开」这类操作，界面上会出现对应的操作确认卡片（追加评论 / 关闭 issue / 重新打开 issue），同样是先展示、确认后才真的调用 GitLab/GitHub API。

### 我的提报

左侧「我的提报」能看到自己提交过的所有 issue 和状态流转：**已提报 → 已领取 → 已合入 → 已关闭 / 被打回**。状态每 10 分钟自动同步一次；标「可验证」的意思是修复已经合到 `test` 分支，可以去测试环境复验了。

## 四、限制 & 常见问题

| 限制 | 值 | 说明 |
|---|---|---|
| 单条消息图片数 | 5 张 | 超过会被拒绝 |
| 单轮工具调用上限 | 30 次 | 到上限会强制收敛给结论，不会无限调查下去 |
| 登录有效期 | 24 小时 | 过期需要重新登录 |
| 仓库自动同步间隔 | 10 分钟 | 也可以在「仓库」页手动触发一次同步 |
| issue 状态轮询间隔 | 10 分钟 | 「我的提报」页看到的状态可能有 10 分钟延迟，不是实时的 |

**它不会做什么**：不会直接改你仓库里的文件，不会自己提交代码/PR。它能做的最大动作就是帮你起草一个 issue 或对已有 issue 做评论/关闭/重开操作——实际改代码还是要人来做。

**问它问题但答非所问/绕圈子**：点 `▼ 不准确`，这是目前唯一的反馈机制。

**看不到某个仓库**：找管理员在「权限」页给自己账号加一条授权。
